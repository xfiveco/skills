# Chisel Architecture And Files

## What Chisel Is

Chisel is a WordPress theme framework that combines:

- Timber and Twig for templating
- a `chisel-scripts` build pipeline on top of modern WordPress tooling
- SCSS organized with ITCSS
- `theme.json` as the design token source of truth
- auto-registration patterns for blocks, assets, CPTs, taxonomies, ACF options, and AJAX routes

The main idea is that `core/` is the framework layer and `custom/` is the project layer.

## How A Chisel Theme Boots

The runtime entry point is `functions.php`.

Boot order in a Chisel project:

1. Composer autoload is loaded.
2. A Chisel autoloader registers both `core/` and `custom/app/`.
3. Timber is initialized.
4. Core singletons are booted:
   - AJAX controller
   - blocks
   - ACF
   - ACF blocks
   - assets
   - comments
   - site
   - sidebars
   - theme
   - custom post types
   - custom taxonomies
   - Twig
   - Gravity Forms integration
   - WooCommerce integration
   - Yoast integration
   - Timber cache
5. `custom/functions.php` boots project-specific singletons from `custom/app/WP/`.

## Namespace And Override Rule

The public docs are correct, but there is one local detail worth remembering:

- classes are namespaced under `Chisel\`
- the custom layer uses namespaces like `Chisel\WP\Custom\Assets`
- the autoloader strips the `Custom` path segment when it resolves files inside `custom/app/`

That means:

- `Chisel\WP\Custom\Assets` maps to `custom/app/WP/Assets.php`
- `Chisel\WP\Custom\Site` maps to `custom/app/WP/Site.php`
- `Chisel\Timber\Custom\ChiselPost` maps to `custom/app/Timber/ChiselPost.php`

This is why the docs talk about "custom namespace" and "same file path as core" at the same time.

## Core vs Custom

Do not modify `core/` unless you explicitly want a framework fork and are willing to lose those changes on `npm run update-chisel`.

Preferred pattern:

- extend or override PHP behavior in `custom/app/`
- override Twig templates in `custom/views/`
- add blocks in `src/blocks/` or `src/blocks-acf/`
- add styles and scripts in `src/`

A typical Chisel project includes these project extension stubs:

- `custom/app/WP/Assets.php`
- `custom/app/WP/CustomPostTypes.php`
- `custom/app/WP/Acf.php`
- `custom/app/WP/Ajax.php`
- `custom/app/WP/Twig.php`
- `custom/app/WP/Site.php`
- `custom/app/Timber/ChiselPost.php`

## Directory Guide

### Framework and integration layer

- `core/`: Chisel framework code. WordPress integration, helper classes, factories, traits, Timber classes, plugin integration.
- `vendor/`: Composer dependencies.
- `node_modules/`: npm dependencies.

### Project extension layer

- `custom/app/`: project PHP extensions and overrides.
- `custom/views/`: Twig overrides. Same file name/path wins over `views/`.
- `custom/functions.php`: boots project-specific singleton classes.

### Rendering layer

- `views/`: base Twig templates, pages, archives, singles, components, partials, WooCommerce Twig templates.
- `patterns/`: editor block patterns.
- PHP template entry files at the theme root like `page.php`, `single.php`, `archive.php`, `search.php`: hand WordPress control over to Twig/Timber.

### Asset source layer

- `src/scripts/`: JS entrypoints and modules.
- `src/styles/`: SCSS files organized by ITCSS layers.
- `src/design/`: design tokens, tools, mixins, settings, theme helpers.
- `src/blocks/`: native Gutenberg blocks.
- `src/blocks-acf/`: ACF blocks, usually with Twig templates and block-local ACF JSON.

### Build and static assets

- `build/`: compiled assets and compiled block outputs.
- `assets/`: static files that are not generated by the JS pipeline, such as fonts, images, icons, hashes.
- `acf-json/`: global ACF field group JSON.
- `theme.json`: WordPress design tokens and editor settings.

## Key Top-Level Files

- `functions.php`: bootstrap and singleton initialization.
- `style.css`: WordPress theme header file.
- `theme.json`: design tokens and editor/runtime design settings.
- `webpack.config.js`: project webpack extension config.
- `package.json`: commands and build pipeline entry points.
- `composer.json`: PHP dependency manifest.
- `phpcs.xml`, `twig_cs.php`, `eslint.config.mjs`, `stylelint.config.mjs`, `prettier.config.mjs`: quality gates.

## Where To Add New Code

### PHP and WordPress behavior

- Custom post types: `custom/app/WP/CustomPostTypes.php` via `chisel_custom_post_types`
- Custom taxonomies: `custom/app/WP/CustomPostTypes.php` via `chisel_custom_taxonomies`
- ACF options pages: `custom/app/WP/Acf.php`
- AJAX routes: `custom/app/WP/Ajax.php`
- Custom AJAX handlers: `custom/app/Ajax/`
- Custom Twig functions and filters: `custom/app/WP/Twig.php`
- Custom Timber post/term class mapping: `custom/app/WP/Site.php`
- Custom Timber classes: `custom/app/Timber/`

### Templates and UI

- Reusable Twig components: `views/components/`
- Template overrides: `custom/views/`
- Template partials: `views/partials/`
- Page/archive/single Twig templates: `views/`
- WooCommerce Twig templates: `views/woocommerce/`

### Frontend assets

- App JS modules: `src/scripts/modules/`
- Editor JS: `src/scripts/editor/`
- SCSS: `src/styles/`
- Design helpers and token functions: `src/design/`
- Static design files: `assets/`

### Editor-driven content building

- Native blocks: `src/blocks/`
- ACF blocks: `src/blocks-acf/`
- Block patterns: `patterns/`
- Global ACF field groups: `acf-json/`

## Template And Page Flow

Typical page flow:

1. WordPress resolves a PHP template entry file such as `page.php` or `single.php`.
2. That file calls `Timber::render(...)`.
3. Twig renders a template in `views/`.
4. `core/WP/Site.php` injects global context such as:
   - logo
   - menus
   - sidebars
   - footer sidebars
   - page title helper output
5. Components are rendered from `views/components/`.
6. If a matching file exists in `custom/views/`, that version can override the default template.

Key template files in a Chisel project:

- `views/base.twig`
- `views/page.twig`
- `views/archive.twig`
- `views/single.twig`
- `views/components/header.twig`
- `views/components/footer.twig`

## Blocks Flow

Chisel's block system is one of the most important parts of the framework.

Implementation in Chisel:

- `core/WP/Blocks.php` sets up block categories, pattern categories, style behavior, and block filters.
- `core/Factories/RegisterBlocks.php` scans the compiled block output and registers blocks.
- Native block source lives in `src/blocks/<block>/`.
- ACF block source lives in `src/blocks-acf/<block>/`.

Important practical detail:

- registration scans the compiled `build/blocks/` and `build/blocks-acf/` folders
- source code lives in `src/blocks/` and `src/blocks-acf/`
- if the build output does not exist, the block will not register

That is why block work nearly always starts with:

- `npm run dev`
- or `npm run build-scripts`

## Design System Flow

The public docs describe `theme.json` as the token source of truth, and this theme matches that approach.

Implementation in Chisel:

- tokens are defined in `theme.json`
- SCSS accesses those tokens through functions in `src/design/tools/_theme.scss`
- styles are then organized through ITCSS layers inside `src/styles/main.scss`

So the intended pattern is:

1. define colors, font sizes, font families, spacing, radii, shadows, and layout in `theme.json`
2. use SCSS helper functions like `get-color()` and `get-font-size()` in the design layer
3. consume those in component and block styles

## Project Setup Expectations

- Chisel projects typically include custom WP extension stubs in `custom/app/WP/`, following the framework's extension pattern.
- You may find an existing override for the post class map in `custom/app/WP/Site.php`.
- A custom Timber post class example often exists in `custom/app/Timber/ChiselPost.php`.
- The command `npm run add-page` is available in Chisel projects for scaffolding new page templates. Run it without parameters to use the interactive prompts.

## Fast Mental Model

When deciding where a change belongs:

- framework behavior extension: `custom/app/`
- template override: `custom/views/`
- template creation: `views/`
- reusable editor content block: `src/blocks/` or `src/blocks-acf/`
- reusable visual pattern: `patterns/`
- styling: `src/styles/`
- token or global design system change: `theme.json` and `src/design/`
- never as a first choice: `core/`
